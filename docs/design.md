# 设计边界文档

## 1. 项目目标

本项目实现一个基于 eBPF 的多协议网络服务原型系统，围绕 **DNS** 与 **gRPC** 两类网络服务，完成以下目标：

1. 对 DNS 与 gRPC 实现实时性能监控。
2. 对 DNS 实现可验证的 eBPF 热点请求加速。
3. 在 Host-VM 虚拟化场景下验证通信路径与性能变化。
4. 设计并实现“内核侧热点缓存 + 用户态完整缓存”的双端缓存机制。

## 2. 核心定位

本项目的核心定位不是通用网络中间件，也不是完整协议栈替代方案，而是一个面向比赛与答辩展示的 **可验证原型系统**。  

系统重点关注的是：

- 能否稳定观测 DNS / gRPC 的性能指标
- 能否对 DNS 热点请求形成明显的内核侧加速效果
- 能否在虚拟化场景中展示 Host 与 VM 之间通信开销下降
- 能否将研究性内容收敛为可交付成果

## 3. 明确范围

### 3.1 本项目要做什么

#### DNS

- 提供一个可运行的 DNS baseline 服务
- 对 DNS 请求进行监控
- 对热点 DNS 请求进行 eBPF 快速路径处理
- 支持用户态缓存回填到内核侧热点缓存

#### gRPC

- 提供一个可运行的 gRPC baseline 服务
- 采集 gRPC 的请求量、延迟、错误率等指标
- 分析虚拟化场景下的通信与性能特征
- 作为“第二类网络服务”的监控样例

#### 双端缓存

- 在 eBPF 侧保留小容量热点缓存
- 在用户态保留更完整的缓存
- 由用户态向内核态回填热点项
- 采用 TTL、访问频率、简单淘汰策略管理缓存内容

#### 虚拟化验证

- 在 Host-VM 场景下部署实验环境
- 通过对比基线与优化后的结果，验证优化收益

### 3.2 本项目不做什么

以下内容明确不纳入本次实现范围：

- 完整的 gRPC 应用层加速
- LDAP 协议支持
- 工业级多节点缓存系统
- 分布式一致性协议
- 复杂机器学习或强化学习调参
- 深度改造 KVM / hypervisor 网络栈
- 复杂 Web Dashboard
- 通用型多协议代理产品化

## 4. 服务选型边界

### 4.1 DNS

DNS 是本项目的主优化对象。原因是：

- 协议结构简单
- 热点请求明显
- 容易做缓存命中验证
- 容易在 eBPF 快速路径中体现收益

DNS 的职责是：

- 作为监控对象
- 作为加速对象
- 作为性能对比基线对象

### 4.2 gRPC

gRPC 是本项目的辅助监控对象。原因是：

- 代表现代微服务通信场景
- 更能体现项目的多协议能力
- 适合做连接、时延、吞吐、错误率分析

gRPC 的职责是：

- 作为监控样例
- 作为虚拟化通信开销分析对象
- 不作为第一阶段快速路径优化对象

## 5. 系统架构边界

系统由四层组成：

1. **基础服务层**
   - DNS 服务
   - gRPC 服务

2. **eBPF 监控层**
   - 请求计数
   - 延迟统计
   - 错误统计
   - 连接与网络指标采集

3. **eBPF 快速路径层**
   - DNS 热点请求内核侧快速处理
   - 命中后直接返回
   - 未命中则回退到用户态

4. **用户态控制与缓存协同层**
   - 加载与卸载 eBPF 程序
   - 读取指标
   - 维护用户态缓存
   - 更新内核侧热点缓存

## 6. 双端缓存边界

### 6.1 内核侧热点缓存

内核侧缓存只承担“最热数据的快速命中”职责，不做复杂逻辑。其特征是：

- 容量小
- 结构简单
- 命中路径短
- 只放最热请求

### 6.2 用户态完整缓存

用户态缓存承担完整缓存管理职责，包括：

- 更多记录的保存
- TTL 管理
- 回源
- 过期处理
- 热点识别与回填

### 6.3 协同规则

协同规则只遵循轻量级策略，不引入复杂控制器：

1. 请求先尝试命中内核侧热点缓存
2. 未命中则交由用户态处理
3. 用户态命中后更新访问统计
4. 热点项回填内核侧缓存
5. 冷门项或过期项被淘汰

## 7. 监控边界

本项目监控的指标只保留能支撑实验结论的最小集合。

### 7.1 DNS 监控指标

- 请求数
- 响应数
- 命中率
- 平均时延
- P95 / P99 时延
- 解析失败数

### 7.2 gRPC 监控指标

- 请求量
- 响应量
- 错误率
- 连接建立情况
- 时延
- 重传或队列相关现象

### 7.3 系统指标

- CPU 占用
- 内核态 / 用户态开销变化
- 上下文切换趋势
- Host-VM 通信路径上的性能变化

## 8. 性能优化边界

本项目只做 **DNS 热点请求加速**，不扩展到所有协议和所有负载。

优化目标仅包括：

- 减少热点 DNS 请求进入用户态的次数
- 减少 Host 与 VM 的重复通信
- 降低热点路径延迟
- 降低 CPU 开销

不做：

- 全协议透明加速
- 全量请求都走内核快速路径
- 面向任意服务的通用代理加速

## 9. 实验边界

### 9.1 基线实验

基线实验用于回答“没有优化时系统表现如何”。

必须完成：

- DNS baseline
- gRPC baseline
- 基线时延、吞吐、CPU 采集

### 9.2 优化实验

优化实验用于回答“eBPF 快速路径和双端缓存是否有效”。

必须完成：

- DNS 热点命中实验
- 命中与未命中对比实验
- Host-VM 场景对比实验

### 9.3 不做的实验

- 多节点分布式压测集群
- 长周期稳定性评估
- 工业级容灾演练
- 复杂流量治理实验

## 10. 交付边界

本项目最终交付物只包含以下内容：

- 项目 README
- 设计文档
- 实验文档
- eBPF 监控与加速原型代码
- DNS / gRPC baseline 服务
- 压测脚本
- 实验结果与图表
- 答辩材料

不包含：

- 生产级发布系统
- 云原生控制台
- 商业化 SLA 保证

## 11. 风险边界

### 11.1 技术风险

- eBPF 程序复杂度过高
- DNS 协议解析出现边界问题
- Host-VM 实验环境不稳定
- 指标采集口径不统一

### 11.2 进度风险

- 试图一次性做完整协议优化
- 监控与加速并行扩张过快
- 论文、比赛、求职任务互相挤占时间

### 11.3 控制原则

- 先跑通基线，再做监控，再做加速
- 先做 DNS，再扩展 gRPC
- 先验证单点收益，再考虑扩展功能
- 先保证可交付，再追求更完整实现

## 12. 下一步执行顺序

1. 完成 DNS baseline 服务
2. 写压测脚本并拿到基线数据
3. 接入最小 eBPF 监控
4. 实现 DNS 热点缓存快速路径
5. 在 Host-VM 场景下验证效果
